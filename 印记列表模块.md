# 任务：实现印记列表模块（前后端联调）

印记列表是登录后的核心主页（路由 `/`）。前端 UI、瀑布流组件与本地 Mock 交互已完成；本模块目标是**将列表数据与展出设置、删除等操作接入后端 API**，替换 `useImprintStore` 中的纯本地 Mock。

## 前置条件

- 登录鉴权模块已完成：`POST /api/auth/login`、`GET /api/auth/me`、`authMiddleware`、前端 Token 拦截器与路由守卫。
- Prisma 已存在 `Memory` 模型（见 `backend/prisma/schema.prisma`），字段与前端 `ImprintListItem` 对齐。
- 首页 UI 与交互逻辑见 `frontend/需求文档/home`；实现入口为 `frontend/src/views/home/`。

---

## 1. 数据模型与字段映射

### 1.1 数据库（Prisma `Memory`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String (cuid) | 印记唯一 ID |
| `userId` | String | 所属用户，列表接口只返回当前登录用户的记录 |
| `title` | String | 标题 |
| `coverUrl` | String | 瀑布流封面图 URL |
| `heightWeight` | Float | 瀑布流高度权重，默认 `1` |
| `isPublic` | Boolean | 公开展示，默认 `false` |
| `linkSuffix` | String @unique | 分享链接后缀，全局唯一，仅字母与数字 |
| `content` / `meta` / `images` | — | 详情页字段，列表接口可不返回或仅详情接口返回 |
| `createdAt` / `updatedAt` | DateTime | 排序与审计 |

### 1.2 前端领域类型（`src/types/imprint.ts`）

列表页使用 `ImprintListItem`：

```ts
interface ImprintListItem {
  id: string
  title: string
  coverUrl: string
  heightWeight: number
  isPublic: boolean
  linkSuffix: string
}
```

### 1.3 API 响应 DTO（建议命名 `MemoryListItemDto`）

与 `ImprintListItem` 字段一一对应；`heightWeight`、`isPublic` 由后端保证类型与默认值，避免前端兜底混乱。

## 2. 后端接口实现 (Backend)

在 `backend/src` 下新增 **`memory`** 模块（与 `auth` 同级），保持 **routes → controller → service** 三层，复用 `authMiddleware`、`asyncHandler`、`AppError`。

### 2.1 目录与文件（新增）

```
backend/src/memory/
├── routes.ts       # 挂载 /api/memories
├── controller.ts   # 参数解析、响应封装
├── service.ts      # 业务逻辑、Prisma 调用
├── schema.ts       # zod 校验
└── types.ts        # DTO、内部类型
```

在 `backend/src/app.ts` 中注册：`app.use('/api/memories', memoryRouter)`。

### 2.2 通用规则

- **鉴权**：除下文「游客可读」接口外，均需 `Authorization: Bearer <token>`；`userId` 从 `req.authUser.userId` 读取。
- **归属校验**：对 `:id` 的写操作，必须先查库并确认 `memory.userId === 当前用户`，否则返回 `404`（不暴露「存在但无权限」）。
- **校验**：请求体使用 `zod`；`linkSuffix` 规则与前端一致：`/^[a-zA-Z0-9]+$/`，长度建议 `1～32`。
- **唯一性**：更新 `linkSuffix` 时若与其他印记冲突，返回 `409`，文案如「该后缀已被占用」。
- **列表排序**：`createdAt` 降序（最新在前）。
- **搜索**：首版可在前端对标题做模糊过滤（与现有 `useImprintStore.filteredItems` 一致）；后端 `GET` 可选支持 `?q=` 查询参数，非必须。

### 2.3 接口列表

| 方法 | 路径 | 鉴权 | 说明 |
|------|------|------|------|
| `GET` | `/api/memories` | 是 | 当前用户的印记列表 |
| `GET` | `/api/memories/:id` | 是 | 单条详情（含 `content`、`meta`、`images` 等，供编辑/详情页复用） |
| `PATCH` | `/api/memories/:id` | 是 | 更新展出设置：`isPublic`、`linkSuffix`（至少传其一） |
| `DELETE` | `/api/memories/:id` | 是 | 删除印记（级联由 Prisma 处理） |

**本阶段不包含**（留给「发布/编辑印记」模块）：

- `POST /api/memories` 创建印记
- 封面上传、正文与多图保存

### 2.4 请求 / 响应约定

**`GET /api/memories`**

- 响应：`{ items: MemoryListItemDto[] }`
- 无分页要求；数据量增大后再加 `cursor` / `page`（TODO 注释占位即可）。

**`PATCH /api/memories/:id`**

- Body（zod）示例：

```ts
{
  isPublic?: boolean
  linkSuffix?: string  // 仅字母数字，更新时校验全局唯一
}
```

- 响应：更新后的 `MemoryListItemDto`。

**`DELETE /api/memories/:id`**

- 响应：`204 No Content` 或 `{ success: true }`（与项目现有 JSON 风格统一即可）。

**`images` 字段**：库内为 JSON 字符串；列表接口不返回；详情接口 `JSON.parse` 后以 `string[]` 返回，解析失败则 `[]`。

### 2.5 错误码约定

| 场景 | HTTP | code 示例 |
|------|------|-----------|
| 未登录 / Token 无效 | 401 | `UNAUTHORIZED` |
| 印记不存在或非本人 | 404 | `NOT_FOUND` |
| `linkSuffix` 格式非法 | 400 | `BAD_REQUEST` |
| `linkSuffix` 已被占用 | 409 | `CONFLICT` |

## 4. 页面功能与交互（UI 已实现，本模块对接数据）

路由：`/`（`views/home/index.vue`），需登录（`meta.requiresAuth: true`）。

### 4.1 布局（自上而下）

1. **顶部操作区**（`search-header`）：有数据时吸顶；列表为空时隐藏。左侧搜索「搜索印记」，右侧 `+` 跳转发布页 `/publish`。
2. **双列瀑布流**（`waterfall-grid` + `imprint-card`）：封面比例自适应 + 标题单行省略 + `...` 更多菜单。
3. **空状态**：无数据时展示 `imprint-state` 文案与「+ 封存新印记」按钮（同跳转 `/publish`）。
4. **底部浮动 Tab**（`floating-tab-bar`）：左主页高亮，右跳转 `/profile`。

### 4.2 交互链路

| 操作 | 行为 | 本模块后端 |
|------|------|------------|
| 搜索 | 标题模糊过滤（前端 computed） | 可选 `?q=`，非必须 |
| 点击卡片主体 | 跳转 `/article/:id` 预览 | 详情数据可仍用 Mock 直至详情模块联调 |
| 点击 `...` | 底部 Action Sheet，不冒泡到卡片点击 | — |
| 复制感应链接 | 剪贴板 + Toast「复制成功」 | 仅用本地 `id` + `linkSuffix` 拼链 |
| 展出设置 | 半屏弹窗改 `isPublic` / `linkSuffix` | **PATCH** |
| 编辑印记 | 跳转 `/publish?id=` | 后续模块 |
| 删除 | 二次确认后移除 | **DELETE** |

### 4.3 展出设置弹窗（`exhibit-settings-sheet`）

- 开关：绑定 `isPublic`。
- 后缀输入：前缀 `/m/{id}-` 只读，用户仅改后缀；输入时 `sanitizeLinkSuffix`，保存前 `isValidLinkSuffix`。
- 警告文案固定展示：「⚠️ 修改后缀后，原感应链接将失效，必须重新使用 NFC 工具写入实体模型」。
- 主按钮「保存并复制新链接」：PATCH 成功后关闭弹窗、复制新链接、Toast「设置已保存，新链接已复制」。

## 6. 验收标准 (DoD)

1. 登录用户进入首页，自动加载**本人**印记列表；未登录被重定向到 `/login`。
2. 列表为空时显示空状态，不显示顶部搜索栏；有数据时瀑布流与搜索正常。
3. 搜索框输入后，仅前端按标题过滤，结果实时更新。
4. 「展出设置」保存后，数据库 `isPublic` / `linkSuffix` 更新；新链接可复制；后缀冲突有明确错误提示。
5. 删除印记后列表刷新且数据库记录消失；取消删除无变化。
6. 用户 A 无法 PATCH/DELETE 用户 B 的印记（返回 404）。
7. `linkSuffix` 含特殊字符或为空时，前后端均拒绝保存。

## 7. 开发原则

- **先后端、后前端**：先实现 `GET` / `PATCH` / `DELETE` 并用 curl 或 REST Client 验证，再改 Pinia 与 hooks。
- **模块边界**：列表模块只做「读列表 + 展出设置 + 删除」；创建/编辑正文与图片归「发布印记」模块。
- **类型对齐**：后端 DTO 与 `ImprintListItem` 保持一致，避免字段改名导致瀑布流组件大面积改动。
- **前端结构**：遵循 `views/home/index.vue` + `hooks/index.ts`；组件继续放在 `components/imprint/`；UI 使用 **tdesign-mobile-vue**。
- **渐进式**：详情页 `/article/:id` 可暂时仍读 Mock；列表联调完成后再做详情 API。

## 8. 实施顺序（建议）

1. 后端 `memory` 模块：`GET` 列表 → `PATCH` 展出 → `DELETE`。
2. 前端 `api/memories.ts` + `imprint` store 拉取列表。

4. 手工联调：注册 → 登录 → 用 Prisma Studio 插入测试 `Memory` → 首页验证列表与操作。
5. （后续）发布模块实现 `POST /api/memories` 后，空状态「封存新印记」可走真实创建流程。
