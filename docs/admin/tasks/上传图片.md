# 任务：前端图片上传组件集成纯前端压缩逻辑

在「新增/编辑印记页」的图片上传逻辑中，为了实现后续的【前端直传 OSS】方案，我们需要先在本地完成图片的极速压缩，并返回标准的二进制文件对象。

## 1. 依赖安装

请在 `frontend` 目录下，安装 `compressorjs`：
`npm install compressorjs`

## 2. 核心压缩逻辑封装 (基于 Promise)

请在 `frontend/src/utils/` 下新建 `imageCompressor.ts`，封装一个 `compressImage(file: File): Promise<File | Blob>` 函数。
具体要求：

1. **防 OOM 拦截**：如果 `file.size > 50 * 1024 * 1024` (50MB)，直接 `reject(new Error('图片体积过大，请选择 50MB 以内的照片'))`。
2. **Compressor 配置**：
   - `quality: 0.6`
   - `maxWidth: 1920`, `maxHeight: 1920`
   - `mimeType: 'image/jpeg'` (防 PNG 负压缩，强制转 JPEG)
3. **返回值要求 (核心)**：
   - 压缩成功后，`success` 回调中会得到一个 `result`。
   - 请直接 `resolve(result)`。因为为了后续前端直传 OSS，我们需要保留原始的 `File` 或 `Blob` 二进制对象，**切勿将其转换为 Base64 字符串**。

## 3. TDesign <t-upload> 拦截与接入

请修改 Editor 页面中的上传逻辑：

1. 使用 TDesign `<t-upload>` 的自定义上传方法（如 `requestMethod` 或重写默认上传事件）。
2. 当获取到用户选择的文件列表后，循环调用 `compressImage`。
3. 为了模拟后续真实的直传体验，当前阶段请在拿到压缩后的 `File/Blob` 后，`console.log` 打印出压缩前后的文件体积对比（以 KB 为单位），验证压缩是否成功。
4. 在页面 UI 上维护一个 Loading 状态，避免在压缩期间用户进行其他操作。
